# BK1.2 + BK2 Backup — Coded (commit c3a60cc, 2026-06-01)

**Fact:** BK1.1 was already done (SupabaseBackupService + BackupPort + tests from earlier session). This session added BK1.2 (periodic trigger) + BK2 (restore gate + Settings UI).

**Why:** P3_BACKUP_PLAN.md requirement. Protects learner progress on device switch.

## What's coded

**BK1.2 — Periodic backup:**
- `PerformBackupUseCase` (`application/backup/`) — orchestrates `shouldBackup(lastMs, nowMs)` → `backupNow(userId)` → persist timestamp. 8 unit tests.
- `AsyncStorageAdapter.getLastBackupAtMs/setLastBackupAtMs` — device-level (NOT user_stats); restored backups don't reset the per-device 6h throttle.
- Wired in `container.ts` `session.start()` — every foreground triggers the 6h check.
- `services.backup.triggerIfNeeded(nowMs)` exposed for quiz-completion callers.
- Analytics: `backup_completed`, `backup_failed`.

**BK2 — Restore on device switch:**
- Hydration gate in `createContainer()` BEFORE `openDatabase()`: `auth.getSession()` → if session AND no local user.db → `backupService.restore(userId)` → THEN open + migrate.
- **Critical ordering preserved:** restore → openDatabase → migrations. Clobbering a migrated-empty-schema with a backup is impossible.
- `services.backup.forceRestore()` — for manual trigger from Settings (no `checkLocalDbExists` guard; user explicitly confirmed overwrite).
- Settings screen: "Restore from backup" button in Account card (signed-in only) → confirmation modal → ok/no_backup/error result display.

## Key deviations from plan

- `last_backup_ms` in AsyncStorage, NOT user_stats migration — restores don't reset per-device throttle.
- No `RestoreBackupUseCase` class — logic is inline in `createContainer()` (composition root doesn't need a testable class wrapper; acceptance is manual device test not unit test).
- `PerformBackupUseCase` named differently from plan's `ScheduleBackupUseCase` — name better reflects its role (orchestrates; not just schedules).

## How to apply

When a user reports "I installed on a new device and lost my progress", the fix path is:
1. Verify `user-backups` bucket exists and RLS is configured
2. Verify `SupabaseBackupService` env vars set (`EXPO_PUBLIC_SUPABASE_URL`, `EXPO_PUBLIC_SUPABASE_ANON_KEY`)
3. User: sign in → hydration gate fires automatically on next cold launch

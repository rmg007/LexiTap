# AGENTS.md — LexiTap

LexiTap is a solo-founder, offline-first ESL vocabulary mobile app. Global ESL learners (non-native English speakers) only — American-student vocab is a separate future "USA Schools" product. Never blend audiences.

## Repo layout

| Path | What |
|------|------|
| `content-tool/` | Track A. Node + TS CLI that builds the bundled read-only `words.db`. |
| `mobile/` | Track B. Expo + React Native app. Source of truth = local SQLite. |
| `lexitap-docs/` | Full research/product documentation (8 categories). The single canonical doc layer. |
| `memory/ docs/ plans/` | Project memory, ADRs, and implementation plans. |

## Architecture (non-negotiable)

Hexagonal / clean. Dependencies point inward only:
`presentation -> application -> domain` and `infrastructure -> domain`.

- `domain/` — pure TS. NO imports from react, react-native, expo-sqlite, @supabase/*. SRS scheduling lives here.
- `application/` — use cases; entitlement/paywall logic here.
- `infrastructure/` — the ONLY place raw SQL and expo-sqlite/supabase/IAP imports live.
- `presentation/` — RN screens + the 4 assessment widgets. LexiTap-specific.

Two databases: `words.db` (read-only, bundled, built by Track A) + `user.db` (read-write, on device), joined via `ATTACH`. Cloud (Supabase) holds auth + content-error reports + encrypted `user.db` blob backups in Supabase Storage. Device is always authoritative; cloud is never queried for live state.

## Hard rules

- TypeScript strict; `noUncheckedIndexedAccess` on. No `any` (use `unknown` + guards).
- Named exports only. Absolute imports via `@/` → `src/*`. No `lodash`.
- No `TextInput` in `mobile/src/presentation/screens/QuizScreen.tsx`, any future `mobile/src/presentation/screens/quiz/`, or `mobile/src/presentation/components/assessments/` — passive recognition UX (tap/drag/match/classify) only.
- No emoji in the app UI. Use the `Icon` component ([`src/presentation/components/Icon.tsx`](mobile/src/presentation/components/Icon.tsx) — a lean hand-curated registry of authentic Lucide glyphs rendered via `react-native-svg`; add a glyph by pasting real Lucide `<path>`/`<rect>` data, never hand-draw) or plain text. Emoji render inconsistently across platforms and fail the Figma "emoji 0" gate. **Enforced** by `guardrails.mjs` (PreToolUse) for non-test `mobile/(src|app)` `.ts(x)` — pictographic ranges only; the `→ ↔ ─` comment chars are exempt. `Icon` is a native (`react-native-svg`) dependency → it reaches device only via a new EAS build, not a JS-only EAS Update.
- Parameterized SQL only, in named query functions under `infrastructure/db/`. Never interpolate.
- Active-word queries filter `deleted_at IS NULL`; history/replay queries deliberately do not.
- New-word learn batches (`selectNewWords`) order by `difficulty ASC, frequency_rank ASC` — easiest-first, common-as-tiebreak. **Never order the learn batch by `created_at`** (content-pipeline insertion order = noise, mixes trivial and hard words). Not `cefr_level` either until it's backfilled — only 58/2848 foundation words carry a CEFR level today, so it sorts as ~all-NULL; `difficulty` (1–4) has 100% foundation coverage and is the usable difficulty signal. Revisit CEFR ordering once the backfill lands.
- `quiz_attempts` and `event_log` are append-only — compensating inserts, never UPDATE/DELETE. `event_log` is scoped to the offline pending-writes buffer (e.g. `content_errors` awaiting sync), not a general analytics sink — wire to PostHog/Amplitude if analytics is needed.
- Every SRS write tags `scheduler_version`. SRS v1-fixed intervals: +1/3/7/14/30d by mastery 0-5.
- Streak boundaries evaluated in the user's IANA timezone, never UTC. No `new Date()` for streak comparison.
- Secrets: `.env` in dev, EAS secrets in prod. Never commit secrets or hardcode them. `OPENAI_API_KEY` (image-gen tooling) is dev/build-only — never bundled into mobile.
- RN/Expo conventions (lists, images, animation, styling, the `react`/Metro fragility): read [`mobile/EXPO_NOTES.md`](mobile/EXPO_NOTES.md) before touching rendering/styling. Notably: keep the `domain/index.ts` barrel (decided), `expo-image` not RN `Image`, FlashList for long lists, nativewind stays (unistyles rejected).
- Visual assets (designs, CSS, images, icons): follow [`scripts/README.md`](scripts/README.md) — right tool, canonical home, CRUD rules. Edit sources (SVG/tokens/Figma), regenerate derivatives. Final store icon + primary logo need Ryan's sign-off and ship as vectors (never AI-generated PNGs); everything else (og/marketing/content imagery) generate freely.
- Crash reporting (Sentry) imports live ONLY in `infrastructure/crash/`. Every event passes the `beforeSend`/`beforeBreadcrumb` PII scrub (fail-closed: scrub throws → drop event); never `Sentry.setUser` with email/id; drop `sync` + network breadcrumbs; no tracing/replay/screenshots. Env-gated by `EXPO_PUBLIC_SENTRY_DSN` (inert if unset).
- IAP port contract (2026-06-10): ONE port — domain `IapPort` (`@/domain/iap/IapPort`); the legacy `infrastructure/iap/IapService.ts` duplicate was deleted, never re-add it. `restorePurchases()` returns `null` = restore FAILED (unconfigured SDK/network) vs `[]` = succeeded-and-owns-nothing — UI must never show "no purchases found" for `null` (a fallback value that overlaps a legitimate result hides failures). `logIn`/`logOut` resolve booleans, never throw; the container's alias dedup (`syncIapIdentity`) commits `lastAliasedUserId` only on success so failures stay retryable; RevenueCat `LOG_OUT_ANONYMOUS_USER_ERROR` counts as logOut success. The `react-native-purchases` jest mock's `PURCHASES_ERROR_CODE` values must mirror the REAL SDK enum (`@revenuecat/purchases-typescript-internal/dist/errors.d.ts`) — they had silently diverged.
- Native sign-in (AUTH-1): `cancelled` AuthErrorKind = user dismissed the OS sheet → UI shows NO error; `unavailable` → hide the entry point. Do NOT route non-OTP auth failures through the shared `mapError` 4xx branch — it emits "That code is invalid or has expired." (the trap `deleteAccount` and `signInWithIdToken` both dodge explicitly). The native `AppleAuthenticationButton` has no `disabled` prop — gate via wrapper `pointerEvents` + a self-guarding handler.
- Restoring a backup must NEVER overwrite the live `user.db` while the SQLite connection is open — the open connection's stale page cache can flush over the restored file (data loss/corruption). expo-sqlite here uses the default DELETE journal (no `-wal`/`-shm` sidecars). Two restore seams: `BackupPort.restore` writes `user.db` directly and is safe ONLY before `openDatabase()` (the BK2 boot hydration gate). After boot (the Settings "restore from backup" flow) use `BackupPort.stageRestore` → downloads to a staging file beside `user.db` + arm `AsyncStorageAdapter.setPendingRestore()`; `container.applyPendingRestore()` promotes the staged file at the NEXT launch, before `openDatabase()`. Single source of truth for both paths: `infrastructure/backup/userDbPath.ts`.

## content-tool invariants

- **Input format is JSONL, not CSV. Do not import from CSV files.** `data/input/words_master.jsonl` is the single source of truth (one JSON object per line, one word per line). Load it with `import --source data/input/words_master.jsonl` (a `.jsonl` source auto-routes to `import-master`; no `--tier` flag — `categories` carries CEFR + tiers). The legacy per-tier CSV `import` path survives ONLY for the `export` self-bootstrap and `import-pseudo`; never use the old `foundation.csv`/`toefl.csv` stubs as content sources. Generate/refresh the master with `export-master` (DB → JSONL round-trip). See [`plans/CONTENT_PIPELINE_JSONL_PLAN.md`](plans/CONTENT_PIPELINE_JSONL_PLAN.md).
- **`import-master` is a full upsert; children are clean-slate; whole-word removal is pruned by default.** Per word: word fields + `cefr_level` + `reviewed` + synonyms/antonyms are overwritten, and `word_tiers` / `word_senses` / `sense_examples` / `word_questions` are deleted-then-reinserted. The master file is authoritative — removing a tier/sense/question from a line removes it from the DB on re-import, **and removing a word's line entirely soft-deletes its `words` row** (`deleted_at`) plus hard-deletes its `word_tiers`/`word_senses`/`sense_examples`/`word_questions`. This closes a real recurring bug: 161 "purged" function words (2026-07-05) and 27 removed proper nouns (2026-07-07) both zombied back into a shipped `words.db` a day later because the old upsert-only importer never noticed a word had vanished from `words_master.jsonl` (`memory/2026-07-07_proper-noun-purge-and-zombie-words-bug.md`). Pruning is the default because every real invocation always passes the whole canonical `words_master.jsonl`, never a filtered subset — pass `--no-prune` (CLI) / `{ prune: false }` (`importMaster()`) to import a deliberately partial file without touching anything absent from it; `--dry-run` previews the prune set (`findPruneCandidates`) without writing. Word IDs stay `makeWordId(word)` (surface-form only), so re-adding a previously-pruned word is a plain upsert that clears `deleted_at` — review history survives a rebuild.
- **`categories` array is the single place for CEFR level + tier membership.** Each word has `"categories": ["B2", "foundation", "toefl"]`. The import pipeline routes CEFR values (A1/A2/B1/B2/C1/C2) → `words.cefr_level` and all other slugs → `word_tiers`. Never set `cefr_level` directly on a word object; never use a separate `tiers` field.
- **`reviewed` is a first-class field on every word.** `words.reviewed INTEGER DEFAULT 0` in the DB; `"reviewed": false` in JSONL. It is Ryan's QA flag (set true after definition/senses/questions/audio are checked). Never omit it; never strip it during import/export.
- **Senses live inside the master JSONL, not in a separate file.** `"senses": []` for un-enriched words. The enrichment run writes senses back in-place. `ingest-senses` as a separate command is merged into `import` — do not resurrect it as a standalone step.
- **Questions live inside the master JSONL alongside senses.** `"questions": []` for un-enriched words; 5 questions per word for initial seed. All question types are click or drag-and-drop — **no typing, ever** (passive-recognition constraint). Types: `multiple_choice`, `definition_match`, `fill_blank`, `sentence_order`, `true_false`. Each question has `hint` (nullable — shown on request before answering) and `explanation` (nullable — shown post-answer). `word_questions` table in `words.db`; `quiz_attempts.question_id` (nullable) in `user.db` links attempts to authored questions.
- **`quiz_attempts.question_id` is nullable by design.** `NULL` = algorithmic question (legacy). Populated = authored question from `word_questions`. Never make it NOT NULL — old attempts must remain valid.
- **2,848 foundation words + 1,837 TOEFL words is the current scope (2026-07-06) — do not add new word lists without Ryan's approval.** Seeding is expensive even at the cheap end (a `gpt-4o-mini` Batch/sync run is ~$0.001/word for senses+questions, ~2x that for a bare word needing base fields too). Adding a new word list before existing ones are fully seeded dilutes the effort. When ingesting a new bare-word list: (1) dedup + lemmatize + drop function/multiword/nonword entries against the existing corpus first (free, offline — `wordfreq`/`simplemma` work well); (2) a frequency/rarity filter alone is NOT a content-safety filter — words at the frequency threshold can still be inappropriate for a 13+ app (e.g. explicit-content terms hiding at the edge of "common enough"); run a dedicated appropriateness pass, not just an obscurity cutoff.
- **A bare word (no definition yet) needs a non-empty sentinel, never `""`.** `parseMasterFile`/`coerceMasterWord` (`import-master.ts`) hard-require non-empty `definition`/`example_sentence` on every line — an empty string throws and blocks the ENTIRE file from loading (every command that touches `words_master.jsonl` breaks, not just the new words). Use `PENDING_DEFINITION`/`PENDING_EXAMPLE_SENTENCE` (`master-store.ts`) as the placeholder. `enrich-master` recognizes `definition === PENDING_DEFINITION` as `needs_base: true` in its prompt and generates `pos`/`definition`/`example_sentence`/`word_type`/`theme` for that word in the same call as its senses/questions (`validateEnrichBaseFields` gates them before write-back). **Before shipping, strip any word still showing the sentinel** — it satisfies the parser's non-empty check but is garbage content if it reaches a real user; hold it out to a side file for a future enrichment pass instead.
- **A new word's `frequency_rank` must never be left `NULL`.** SQLite `ORDER BY frequency_rank ASC` sorts `NULL` **first**, not last — an un-ranked new word would jump to the front of `selectNewWords`'s difficulty-then-frequency queue, ahead of every common foundation word (regression of the 2026-07-05 learn-batch-ordering fix). Assign a provisional rank at ingest time (e.g. from `wordfreq` zipf score, offset above the corpus's current max) so new words sort sanely until a real frequency source backfills them.
- **Every category slug in `words_master.jsonl` must be registered in `lexitap.config.json`'s `tiers` array**, or `readMasterRecords`/`tierSlugs()` throws and blocks every command that loads the master file. This bit before: the `diagnostic` tier (49 words, added 2026-07-05) was never added to the config, and the gap went undetected for a month because nothing had re-run `enrich-master`/`categorize` since. If you add a new tier tag to any word, add it to `lexitap.config.json` in the same change.
- **Phase 3 + 4 enrichment runs on the OpenAI key, not Anthropic.** The repo has `OPENAI_API_KEY` (root `.env`); there is no `ANTHROPIC_API_KEY`. The Anthropic `enrich-senses` driver survives as legacy but is NOT the live path. Live commands: `categorize` (Phase 3 — CEFR + specialty tiers per word) and `enrich-master` (Phase 4 — senses + 5 questions per word). Both edit `words_master.jsonl` in place, are resume-safe (sidecar `<master>.categorize-done.jsonl` / `<master>.enrich-skipped.jsonl`), require `--limit`, print a cost estimate, and support `--dry-run`. Shared infra: `providers/openaiClient.ts` (fetch-based, no SDK — uses global `fetch`), `commands/master-store.ts` (master IO + resume), `commands/question-validators.ts`.
- **`categorize` MERGES categories, never removes them.** The model's CEFR overwrites a word's CEFR (it is the Phase-3 authority); specialty tiers are ADDED on top of existing membership. `foundation` and any pre-existing tier always survive. It never emits `foundation` and drops unknown/duplicate slugs. Re-run is idempotent via the done-sidecar.
- **`enrich-master` fail-closes on invalid content.** Each generated item must pass identity check + senses V1–V10 + exactly 5 questions Q1–Q9, else it is dropped (logged) and the word stays un-enriched (retry-eligible). Questions are validated by `validateMasterQuestions` — distractor shape is per-type (≥2 for choice types; `true_false` = correct ∈ {True,False} + distractors `[opposite]`; `sentence_order` = empty distractors + ≥3-word sentence; `fill_blank` prompt needs a blank marker), correct must never leak into distractors, every question needs an `explanation`. Generated content is always `reviewed: 0`.
- **`loadSenses` / `loadSenseExamples` must honour the tier filter.** Both functions accept an optional `tier?: string`; when set, they JOIN through `word_tiers` to return only senses/examples belonging to words in that tier. Passing `undefined` (no tier) returns all rows. Callers in `runValidate` must forward `options.tier`. Callers in `buildOutputDb` use the already-scoped `senses`/`senseExamples` vars and do NOT filter by tier again.
- **`buildOutputDb` must validate senses before writing the output DB.** After the existing `validateRows` + error check, call `validateSenseRows(senses, senseExamples, wordIds, { strict })` and throw on errors. Malformed senses must never silently ship.
- **`ingest-senses` must check word_id existence before writing.** FK constraints are disabled (`PRAGMA foreign_keys = OFF` in `openWorkingDb`). Add an explicit `SELECT 1 FROM words WHERE id = ? AND deleted_at IS NULL` guard inside the transaction and throw a descriptive error if the word is absent. Orphan senses are silent data corruption.
- **`INSERT_SENSE` / `INSERT_EXAMPLE` must be plain `INSERT`, no `ON CONFLICT`.** The ingest transaction does a clean-slate delete before inserting, so no conflict can occur. Dead `ON CONFLICT DO UPDATE` clauses are misleading and mask logic errors — remove them.
- **S6 is subsumed by S2 — do not add it back.** S2's contiguity loop at `i === 0` already catches `sense_index !== 0`. A separate S6 check fires twice for the same row. The S2 message at `i === 0` should read "senses must start at sense_index 0, found N" to be self-explanatory.
- **`isGlossStyle` regex must cover common dictionary-gloss openers beyond "word that/meaning".** Patterns like "a person who", "the act of", "the state of", "the quality of", "the type of" are also gloss style. The S8 warning is only useful if the regex is broad enough to catch common offenders.
- **Motion + Reduce Motion (2026-07-06):** `useMotion().spring(preset)`/`.timing(token)` already return `{duration:0}` under the OS Reduce Motion setting — do NOT add a manual `!reduceMotion` guard around a hook-routed animation, it's already handled. A manual guard IS required only for a RAW `withSpring(springs.x)`/`withTiming` call that bypasses the hook (e.g. `SessionCompleteScreen`'s check-icon settle) — `withSpring` does not interpret `{duration:0}` as "jump instantly," so those call sites mount at the final value directly when `reduceMotion` is true, with no animation call at all. New shared presentation primitives from the same session: `usePressScale` (shared press-scale hook, extracted from `Button`), `Card` `interactive`/`onPress` (tappable-card variant, enforces 48px), `Screen` `footer` prop (sticky-footer slot, measured via `onLayout`), `ListRow`/`EmptyState`/`SectionHeader`/`DailyCapMeter`/`KnowledgeMapBar` (`presentation/components/`) — see [`lexitap-docs/03-ux-design/DESIGN_SYSTEM.md`](lexitap-docs/03-ux-design/DESIGN_SYSTEM.md) Component Library for what each renders; reuse them, don't hand-roll a duplicate (`LearnQuickCheckScreen` had silently forked `FeedbackLayer`'s copy banks this same session — exactly this failure mode).

## Done means

`npm run check` (lint + typecheck + test) passes in the affected project **— and would still pass in CI on a clean install.** Local green ≠ CI green.

- **Adding an import means adding a dependency.** When you `import` a package, it MUST land in that sub-project's `package.json` (`npm install --save …`, run *inside* `mobile/` or `content-tool/`). A package that exists only in local `node_modules` (hoisted / once-installed without `--save`) passes local `npm run check` but fails CI's clean `npm ci` with `Cannot find module …`. This is the repeated **"false-green-handoff"** failure mode — it stayed red & invisible for over a week (the missing `@anthropic-ai/sdk`, 2026-06-09).
- **Catch it before pushing:** `cd <subproject> && npm ls 2>&1 | grep -iE 'extraneous|UNMET|missing'` — any hit is a manifest gap. An empty result is the real "done".
- **CI runs on `main` via THREE path-filtered workflows** (since 2026-06-11): `ci.yml` (`mobile/**` + `content-tool/**`), `supabase-ci.yml` (`supabase/**` — Deno tests for Edge Functions), `infra.yml` (`.claude/**` + `scripts/**` + `website/**` — hook syntax, guardrails self-test, `_redirects` loop check). After a push that touches covered paths, confirm green: `gh run watch $(gh run list --workflow=<workflow>.yml --branch main --limit 1 --json databaseId --jq '.[0].databaseId') --exit-status`. **A handoff that claims "tests pass" without CI confirmation is not trustworthy** — verify, don't assert. Two standing rules: **a new test file must get CI coverage in the same commit** (the delete-account Deno tests sat unrun by any CI for a day — 9963f33); **a new or edited workflow must be proven by one green run before anyone relies on it** (deploy.yml sat broken-as-written for weeks with zero runs — wrong env var name, wrong npx package, masked auth failure — deleted 2026-06-11; resurrect from plans/RELEASE_PLAN.md item 16 only with a green workflow_dispatch run first). Paths filters are workflow-level, not job-level — new coverage areas get their own workflow file, never bolted onto ci.yml's filter.
- **If you rename or move anything a config file references** (branches, paths, script names), grep the repo for the old name and update it. The `master → main` rename silently broke CI for weeks because `ci.yml` still triggered on `[master]`.
- **Gitignored *generated* files are a CI failure class.** `mobile/expo-env.d.ts` (generated, gitignored per Expo convention) carries the CSS-module type declarations; without it `tsc --noEmit` fails TS2882 on `import '../global.css'` in every clean checkout. The committed twin `mobile/expo-types.d.ts` exists solely to fix this — **do not delete it.** This kept CI red from the SDK-56 upgrade (`556606c`) until `c188bb9`. General rule: **a workaround you need in a fresh worktree is a failure CI is already having** — check `gh run list` whenever a clean checkout needs manual setup.
- **Piping check output eats exit codes.** `npm run check | tail` reports exit 0 on a FAILED check (the pipeline's status is tail's). Use `set -o pipefail` (or no pipe) when the exit code is the verdict — a typecheck failure slipped past exactly this way (2026-06-10).
